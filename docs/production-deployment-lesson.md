# Урок: безопасный production deploy LandSearch

Цель урока — выпускать только проверенную версию и не переносить случайные изменения из рабочего дерева на production.

## Что произошло

Ошибка возникла потому, что production-сборка была создана из рабочего дерева с накопленными незакоммиченными изменениями. В релиз попали изменения, которые не относились к задаче коротких кадастровых номеров.

Правило: один deploy должен содержать одну понятную задачу. Нельзя собирать production из каталога, где одновременно находятся незавершённые изменения разных задач.

## Правильный процесс

### 1. Зафиксировать границы задачи

Перед изменением записать:

- какие файлы можно менять;
- какое поведение должно измениться;
- какое поведение нельзя менять;
- как будет проверен результат.

Для задачи подписей границы были такими: `frontend/lib/plot-map-layers.ts` и тест слоя. Панели, фильтры, авторизацию и API менять было нельзя.

### 2. Проверить состояние исходников

```bash
git status --short
git diff --name-only
git diff -- frontend/lib/plot-map-layers.ts
```

Если есть чужие или несвязанные изменения, не собирать из этого каталога. Создать отдельную ветку от стабильного коммита или использовать чистый worktree.

### 3. Сначала тесты, потом сборка

```bash
cd frontend
npm test

NODE_ENV=production \
NEXT_PUBLIC_SITE_URL=https://v3163460.hosted-by-vdsina.ru \
NEXT_PUBLIC_API_URL=https://v3163460.hosted-by-vdsina.ru/api/v1 \
npm run build
```

До публикации обязательно проверить:

- тесты: без failed;
- build: завершён успешно;
- в сборке есть только ожидаемый слой/компонент;
- `BUILD_ID` создан.

### 4. Сформировать отдельный release

Каждый release должен иметь дату и идентификатор:

```bash
release=/tmp/landsearch-frontend-release-YYYYMMDD-HHMM.tar
tar --format=ustar -C frontend -cf "$release" \
  .next/standalone .next/static .next/BUILD_ID
```

Нельзя копировать отдельные случайные chunk-файлы и нельзя смешивать `.next` от разных сборок.

### 5. Сделать backup на сервере

Перед остановкой сервиса сохранить текущие каталоги:

```bash
stamp=$(date +%Y%m%d-%H%M%S)
mv /root/landsearch/frontend/.next/standalone \
  /root/landsearch/frontend/.next/standalone.predeploy-$stamp
mv /root/landsearch/frontend/.next/static \
  /root/landsearch/frontend/.next/static.predeploy-$stamp
```

Старую версию не удалять до завершения smoke-теста новой.

### 6. Перезапустить и проверить доступность

```bash
systemctl restart landsearch-frontend
systemctl is-active landsearch-frontend
curl -fsS -o /dev/null -w '%{http_code}\n' https://v3163460.hosted-by-vdsina.ru/
curl -fsS -o /dev/null -w '%{http_code}\n' https://v3163460.hosted-by-vdsina.ru/health
```

`200` от Nginx недостаточен: сервис должен быть `active`, а frontend должен отвечать после перезапуска.

## Обязательный браузерный smoke-тест

Проверить в одном браузере:

1. Главная страница: карта и панели видимы.
2. Поиск `16:24:090704:5600`: найден ровно один участок.
3. Клик по участку: pin, полигон и карточка синхронизированы.
4. Масштаб `15`: отображается последний номер кадастра, например `5083`.
5. Плюс, минус, компас, геолокация и 3D работают справа.
6. Переключение подложки не удаляет границы участков.
7. В консоли браузера нет ошибок.
8. На mobile 390×844 нет горизонтального overflow.

Для визуальной проверки сохранять скриншот в `output/playwright/` и просматривать его, а не считать успешным только наличие HTML.

## Быстрый откат

Если карта, панели или поиск работают неправильно:

1. Остановить frontend.
2. Вернуть последний `standalone.predeploy-*` и `static.predeploy-*`.
3. Вернуть соответствующий `BUILD_ID`.
4. Запустить сервис.
5. Повторить health-check и браузерный smoke-тест.

Откат выполняется до новых исправлений. Нельзя пытаться чинить production вслепую поверх уже подозрительной сборки.

## Definition of Done для deploy

- изменение ограничено заявленной задачей;
- нет несвязанных файлов в release;
- frontend tests проходят;
- production build проходит;
- backup создан;
- HTTP и systemd проверки проходят;
- визуальный smoke-тест пройден;
- console errors отсутствуют;
- понятен путь отката.

Если хотя бы один пункт не выполнен, версия не считается production-ready.
